### goat-security design

```
## When to Use

Use when assessing security posture: before deployment, after adding auth/input
handling, when touching secrets/credentials, or as part of a security-focused audit.
Differs from goat-audit: audit = broad quality sweep, security = threat-model-driven,
OWASP-aware, framework-specific verification.

Step 0 — Gather Context
  1. What's the threat model? (user-facing app, internal tool, library, API)
  2. What auth/security boundaries exist?
  3. Any known vulnerabilities to skip?
  4. What framework security features are already in place?

Phase 1 — Threat Surface Scan
  Scan against checklist: input validation, auth boundaries, secret handling,
  injection vectors (SQL/XSS/command/path), dependency CVEs, permission
  escalation, CORS/CSP. Log every finding with file:line.

Phase 2 — Framework-Aware Verification
  For EACH Phase 1 finding, check if the framework already mitigates it.
  Remove false positives. Flag partial/misconfigured mitigation.
  THIS IS THE KEY DIFFERENTIATOR — goat-audit doesn't have this pass.

Phase 3 — Rank by Exploitability
  Critical: exploitable without auth, data loss/exposure
  High: exploitable with auth, privilege escalation
  Medium: requires specific conditions, defense-in-depth gap
  Low: theoretical, mitigated by other controls

Phase 4 — Self-Check
  Same fabrication gate as goat-audit. Re-read file:line refs. Remove
  unverifiable findings. Flag low-confidence.

Present findings. Ask: "Want me to dig deeper?"
Do NOT auto-advance. Do NOT propose fixes.

Constraints:
- MUST gather threat model before scanning (Step 0)
- MUST check framework mitigation for every finding (Phase 2)
- MUST rank by exploitability, not just severity (Phase 3)
- MUST NOT propose fixes — security audit reports only
- MUST NOT flag framework-mitigated issues as vulnerabilities

Chains with: goat-debug (investigate a vulnerability further), goat-review (security-specific PR review)
```

## Phase C: Cross-Cutting Improvements

### Design principle: Conversational by structure, not by bolt-on

Every skill phase should end with **choices**, not just a gate. A gate is binary (yes/no). A conversation offers options that change what happens next. The human drives the direction, not just the pace.

**Current pattern (gate — weak):**
```
Phase 2 → present findings → "Does this look right?" → proceed
```

**Required pattern (conversational — strong):**
```
Phase 2 → present findings → "Want me to:
  (a) dig deeper on finding #3
  (b) check if finding #1 is a false positive
  (c) look at a different area
  (d) proceed to next phase"
```

Apply this to every skill's phase transitions:
- [ ] **goat-debug**: After diagnosis → "Want me to (a) trace deeper, (b) check related code, (c) propose fix, (d) check footguns for similar past issues?"
- [ ] **goat-audit**: After each pass → "Want me to (a) re-examine specific findings, (b) expand scope, (c) narrow scope, (d) proceed to next pass?"
- [ ] **goat-plan**: Already strong — 7 human gates with options. Template for other skills.
- [ ] **goat-test**: After track generation → "Want me to (a) adjust Track 1 focus, (b) refine Track 2 prompts, (c) add Track 3 items, (d) finalize?"
- [ ] **goat-review**: After findings → "Want me to (a) dig into the riskiest change, (b) check a specific file, (c) compare against spec, (d) finalize verdict?"
- [ ] **goat-investigate**: After document → "Want me to (a) go deeper on a component, (b) check a boundary I missed, (c) map a different area, (d) close investigation?"
- [ ] **goat-security**: After findings → "Want me to (a) verify a specific finding against the framework, (b) check a different attack surface, (c) test an edge case, (d) finalize report?"

### Other cross-cutting improvements

- [ ] **Shared severity scale**: Define in CLAUDE.md, reference from all skills: SECURITY > CORRECTNESS > INTEGRATION > PERFORMANCE > STYLE
- [ ] **Learning loop integration**: Every skill ends with "If this run uncovered a lesson or footgun, update the relevant doc before closing"
- [ ] **Skill chaining**: Add "Chains with" footer to each skill:
  - investigate → plan → (implement via execution loop) → test
  - debug → (human approves fix) → (implement) → test (regression)
  - audit → review (findings become checklist)
  - security → review (security-specific PR review)
  - reflect → (updates instruction files)
  - resume → (any skill, continues where left off)
