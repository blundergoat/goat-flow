# Milestone 2.9: Competitive Additions

New skills, infrastructure, and content based on analysis of 33 competitor repos. Validated by 5 independent agent reviews. These additions build on goat-flow's existing moats — they don't replace the core workflow.

## The Structural Gap

```
UNDERSTAND          PLAN            BUILD           VERIFY          LEARN
─────────          ─────           ─────           ──────          ─────
/goat-investigate   /goat-plan      (skill          /goat-test      (footguns.md)
/goat-audit                         chaining,       /goat-review    (lessons.md)
/goat-debug                         not a new       /goat-security  /goat-reflect
/goat-onboard                       skill)          /goat-resume
```

The BUILD column is not a separate skill — it's the execution loop (READ → CLASSIFY → SCOPE → ACT → VERIFY → LOG) guided by skill chaining (M2.8 Phase C). The real gap is **spec artifacts**: goat-plan produces milestones with task checkboxes, but not testable acceptance criteria that goat-review can verify against. "Did we build what we planned?" is currently unanswerable.

## goat-flow Moats (no competitor replicates)

1. Scanner/scoring engine with fragment-based prompt generator
2. Evidence-anchored learning loop (file:line in footguns.md)
3. Hot-path/cold-path instruction architecture (ai/instructions/ split)
4. Cross-reference validation in CI
5. Autonomy tiers with 5-item micro-checklist
6. Multi-agent parity from one framework (3 agents scored by scanner: Claude, Codex, Gemini; Copilot via setup guide + bridge files)

## Phase A: New Skills

### /goat-reflect — Instruction tuning skill (LOW effort)
**Validated by:** Claude Code, RIPER-5, CC-SDD (3 agents)
**Gap:** Learning loop is manual. Competitors auto-extract lessons and update instruction files.
**What it does:** After a session, uses what the user describes + observable signals (git log, changed files, recent lessons.md entries) to identify repeated friction and propose edits to instruction files. Human approves before changes apply.

**Important distinction:** goat-reflect is **instruction tuning**, not learning loop. It edits CLAUDE.md and ai/instructions/ — NOT footguns.md or lessons.md (those require file:line evidence from real incidents). Behavioral observations ("I asked about X three times") don't have codebase evidence and don't belong in the learning loop files.

**Design constraint:** Agents cannot read their own conversation history programmatically. Phase 1 relies on what the user describes in Step 0 plus observable evidence from the repo, NOT conversation transcript access.

- [x] Design skill phases:
  - Step 0: "What went well? What was friction? Any repeated mistakes?"
  - Phase 1: Based on Step 0 answers + observable signals: recent `git log --oneline -20`, changed files, recent `docs/lessons.md` entries, rejected patterns in git stash/reflog. Identify repeated friction, wrong assumptions, missing context.
  - Phase 2: Check current instruction files — what's missing or misleading that caused friction?
  - Phase 3: Propose specific edits to CLAUDE.md, ai/instructions/, or local instruction files
  - Human gate: "Here's what I'd change in your instruction files. Approve?"
  - MUST NOT auto-edit footguns.md or lessons.md (evidence standards)
- [x] Create `workflow/skills/goat-reflect.md` template
- [x] Create `.claude/skills/goat-reflect/SKILL.md`
- [x] Create `.agents/skills/goat-reflect/SKILL.md`
- [x] Create `.github/skills/goat-reflect/SKILL.md` (Copilot CLI uses `.github/skills/`)
- [x] Add to scanner expected skills list

### /goat-onboard — Brownfield codebase onboarding (MEDIUM effort)
**Validated by:** Shinpr Workflows, Claude-Code-Workflows, CC-SDD (3 agents)
**Gap:** goat-flow assumes you're adding the workflow to a project. No help for understanding an unfamiliar codebase.
**What it does:** Systematically maps a codebase: architecture, dependencies, API surface, risk areas. Auto-generates `ai/instructions/` content.

- [x] Design skill phases:
  - Step 0: "What project? What do you need to understand? Any existing docs?"
  - Phase 1: Detect stack, read manifests, map directory structure
  - Phase 2: Read entry points, follow critical paths, identify boundaries
  - Phase 3: Draft `ai/instructions/` content to temporary location (present inline, NOT auto-populate cold path)
  - Phase 4: Verify generated content against actual code (verification gate)
  - Human gate: "Here's what I'd put in your instruction files — which of these are actually useful?" Human selectively moves approved content to `ai/instructions/`.
  - MUST NOT auto-populate cold path — auto-generated context reduces success by ~3% and increases cost 20%+ (goat-flow's own research)
- [x] Create `workflow/skills/goat-onboard.md` template
- [x] Create skill instances (`.claude/skills/`, `.agents/skills/`, `.github/skills/`)
- [x] Integrate with M2.6 cold path creation (human-gated, not automatic)
- [x] Add to scanner expected skills list

### /goat-resume — Session continuation (LOW effort)
**Validated by:** Claude-Code-Workflow, Simone, RIPER-5, Spec-Workflow (4 agents)
**Gap:** Long features span multiple sessions. Human manually pastes handoff context.
**What it does:** Reconstructs working state AND proposes next action. Must do more than just read files — it establishes the agent's mental model and suggests what to work on next.

**Skill justification:** Earns skill status because it proposes the next action (not just summarizes). The compaction hook restores raw state (modified files, task list, constraints). The skill INTERPRETS that state: "Based on where we left off, I recommend X because Y." A hook reads files — the skill makes a recommendation.

- [x] Design skill phases:
  - Phase 1: Read `tasks/handoff.md` + `tasks/todo.md` + recent `git log --oneline -20`
  - Phase 2: Read current milestone file (if one exists in roadmaps/)
  - Phase 3: Identify: what's in-progress, what's blocked, what's next
  - Phase 4: Present: "Here's where we left off: [task]. Modified files: [list]. I recommend working on [next step] because [reason]. Want to proceed, or re-scope?"
  - Human confirms direction before proceeding
  - If no handoff exists: "No prior session context found. What are you working on?"
- [x] Create `workflow/skills/goat-resume.md` template
- [x] Create skill instances (`.claude/skills/`, `.agents/skills/`, `.github/skills/`)
- [x] Add to scanner expected skills list
- [x] Consider compaction hook integration (resume logic after compaction)

### Spec artifacts from goat-plan (HIGH effort) — HIGHEST LEVERAGE

**Validated by:** Spec Kit, CC-SDD, Spec-Workflow, Claude Code gap analysis (4+ agents)

**Why highest leverage:** This is the connective tissue between plan and review. Without spec artifacts, goat-review can check code quality but can't answer "did we build what we planned?" Competitors with spec-driven development (Spec Kit, CC-SDD, Superpowers) can do two-stage review: spec compliance + code quality. goat-flow currently only does the second.

**Why HIGH effort (not LOW):** Touches 3 skills (plan, review, test) + introduces a new artifact format with traceability. Every competitor that does this (Spec Kit, CC-SDD, Simone) treats it as their core feature. The artifact format needs to be right before downstream consumers are built.

**Approach:** M2.9 adds the `requirements-{feature}.md` artifact to goat-plan ONLY. A separate milestone wires goat-review and goat-test to consume it. Get the format right first.

- [x] Design `requirements-{feature}.md` format:
  - Each requirement: ID, description, acceptance test (binary pass/fail), linked milestone task
  - Format must be parseable by goat-review's conditional spec compliance phase (M2.8 forward-compatible)
- [x] Enhance goat-plan Phase 1 to produce this artifact
- [ ] Test: run goat-plan on a real feature, verify requirements are testable
- [x] Document: goat-review auto-activates spec compliance when this file exists (M2.8 wired this)
- [ ] Future (separate milestone): goat-test gains requirement-derived test cases
- [ ] Future (separate milestone): full traceability: requirement → milestone → implementation → test

## Not Adding (validated as not worth it)

| Idea | Why not |
|------|---------|
| Heavy orchestration (Ruflo swarms, Ralph events) | Scope creep — goat-flow is a workflow framework, not an agent platform |
| Skill marketplace/registry | Premature — land `goat-flow init` first |
| Multi-provider model routing | Agent-specific, not a workflow concern |
| Party mode / multi-persona | SBAO in goat-plan handles multi-perspective |
| MCP server mode | Requires runtime shift |
| YOLO mode / autonomous loop | Contradicts safety-first philosophy. Considered but rejected — goat-debug's "no fixes first" gate is the #1 rated feature and YOLO undermines it |
| Translation/i18n | Premature for current adoption |
| Mermaid diagrams | Nice but not core workflow |
