# Milestone 2.6: Local Context — Project Coding Guidelines

**Archetype:** Redesign — replace the removed footgun-heuristic local context checks with project-specific coding guidelines.

## Background

The old local context checks (2.6.1-2.6.3) inferred where local instruction files should exist by counting footgun mentions per directory. Too aggressive — removed in v0.4.0.

Reviewed 6 real projects: all use `.github/instructions/` today, but the files are too file-scoped (e.g., `php.instructions.md`, `strands-rust.instructions.md`). They should be domain-scoped instead (e.g., `backend.md`, `frontend.md`). The vendor-neutral `ai/instructions/` location doesn't exist yet in any project.

## The Pattern

**Hot path / cold path architecture:**

- **Hot path** — `CLAUDE.md`, `AGENTS.md`, `GEMINI.md` — loaded every turn. How the agent should *behave*: execution loop, autonomy tiers, DoD, essential commands. Must stay under 120 lines.
- **Cold path** — `ai/instructions/` — loaded on demand when the task matches. How *this code* works: project-specific coding guidelines, domain conventions, review standards, security constraints. The stuff that doesn't fit in 120 lines but that agents need when working in a particular area.
- **Router** — `ai/README.md` — tells agents which cold-path files to load for which tasks. Precedence order included.

This works in any repo, with any agent, even with no GitHub or Copilot at all. `ai/instructions/` is the local context layer — the project's coding guidelines, organized by domain.

## Structure

```
ai/
├── README.md                    # router — which files to load for which tasks
└── instructions/
    ├── conventions.md                  # universal project contract
    ├── code-review.md           # review-only guidance
    ├── git-commit.md            # commit format, branch naming, PR workflow
    ├── frontend.md              # frontend domain (React, Vue, etc.)
    ├── backend.md               # backend domain (API, DB, etc.)
    ├── security.md              # cross-cutting security constraints
    └── testing.md               # testing conventions
```

### Domain-based, not file-based

**Wrong** (too granular — scoped to file types):
```
.github/instructions/
├── php.instructions.md          # ← language, not domain
├── python.instructions.md       # ← language, not domain
├── twig.instructions.md         # ← framework template, not domain
├── shell.instructions.md        # ← language, not domain
└── strands-rust.instructions.md # ← library, not domain
```

**Right** (domain-scoped — what the agent needs to know for an area of work):
```
ai/instructions/
├── conventions.md                      # ← project-wide contract
├── frontend.md                  # ← all frontend: React + Twig + CSS + client-side JS
├── backend.md                   # ← all backend: PHP + Python + API + DB
├── security.md                  # ← cross-cutting: auth, input validation, secrets
└── testing.md                   # ← cross-cutting: test conventions, coverage, mocking
```

A domain file can mention multiple languages. `backend.md` for ambient-scribe would cover PHP (Symfony) AND Python (FastAPI) because the agent working on the backend needs both.

### `.github/` files

Two types of `.github/` files, serving different purposes:

**Universal (any project with `.git/`):**
```
.github/
└── git-commit-instructions.md           # commit format, branch naming — loaded by tools on commit
```

This is not agent-specific. It's for any tool or human that makes commits. Created by every setup guide if `.git/` exists.

**Copilot-only (optional):**
```
.github/
├── copilot-instructions.md              # Copilot HOT PATH — execution loop, autonomy, DoD (like CLAUDE.md)
└── instructions/                        # bridges to ai/instructions/
    ├── frontend.instructions.md         # applyTo: "src/frontend/**" → references ai/instructions/frontend.md
    └── backend.instructions.md          # applyTo: "src/api/**" → references ai/instructions/backend.md
```

`.github/copilot-instructions.md` is a full hot-path file — same structure as `CLAUDE.md`/`AGENTS.md`/`GEMINI.md` (execution loop, autonomy tiers, DoD, essential commands). NOT a bridge. It's Copilot's instruction file.

`.github/instructions/` bridges reference `ai/instructions/` — no duplication of cold-path content.

## Router: `ai/README.md`

```markdown
# Project Coding Guidelines

Read `instructions/conventions.md` first for every task.

Then load additional files based on the work:

| Task | Load |
|------|------|
| Frontend work | `instructions/frontend.md` |
| Backend work | `instructions/backend.md` |
| Code review | `instructions/code-review.md` |
| Security-sensitive work | `instructions/security.md` |
| Writing tests | `instructions/testing.md` |

Precedence (highest first):
1. security.md (always applies if touching auth/secrets/validation)
2. code-review.md (for review tasks only)
3. domain file (frontend/backend)
4. conventions.md (always loaded)
```

## Standard Files

### Required baseline (scored by scanner)

| File | Purpose | When |
|------|---------|------|
| `ai/README.md` | Router — read this first, tells agents which instruction files to load | Every project |
| `ai/instructions/conventions.md` | Universal project contract: what the repo is, build/test/lint, conventions, boundaries | Every project |
| `ai/instructions/code-review.md` | Review standards, approval checklist, common anti-patterns | Every project |
| `ai/instructions/git-commit.md` | Commit message format, branch naming, PR workflow | Every project with `.git/` |

### Recommended (not scored)

| File | When to create |
|------|---------------|
| `ai/instructions/frontend.md` | When project has a frontend — framework patterns, component conventions, state management |
| `ai/instructions/backend.md` | When project has a backend — handler patterns, DB conventions, validation, error handling |
| `ai/instructions/security.md` | When project handles auth, user input, secrets, or sensitive data |
| `ai/instructions/testing.md` | When project has test suite — test patterns, coverage expectations, mocking conventions |

### What goes where

**In `conventions.md` (universal contract):**
- What the repo is
- Architecture overview
- How to run/build/test
- Coding conventions used everywhere
- Generated files (don't edit these)
- Dangerous things not to touch
- Common commands

**In domain files (only what differs from base):**
- Framework-specific patterns (React hooks, Go handler pattern, Symfony services)
- Domain-specific conventions (SQL query style, API versioning)
- Area-specific safety rules (auth boundaries, migration constraints)
- Can mention multiple languages if they're in the same domain

**Key rule:** if the same rule appears in 2+ files, it should be in `conventions.md` instead.

## Writing Good Instruction Files

- **Concrete and operational** — commands, file paths, do/don't with examples
- **Short** — answer "what do you need to know to safely edit files in this area?" and stop
- **No duplication** — shared rules in conventions.md, overlays only add or tighten
- **Domain-based, not file-based** — `backend.md` covers all backend languages, not one file per language
- **Specific over general**: "Use `sqlc.arg(name)` in queries" not "write clean SQL"
- **Call out boundaries**: "Never edit `internal/db/*.sql.go`" not "be careful with generated code"

## Key Decisions

| Decision | Why |
|----------|-----|
| `ai/instructions/` is source of truth | Vendor-neutral. Works for any agent, any platform. |
| `ai/README.md` as router | Discoverable by convention. Tells agents what to load without magic auto-loading. |
| Domain-based, not file-based | `backend.md` > `php.md` + `python.md`. Agents work on domains, not languages. |
| `.github/instructions/` are bridges, not copies | No duplication. One place to update content. |
| `.github/copilot-instructions.md` only for Copilot | Only created by setup-copilot.md. Other agents don't need it. |
| `README.md` + `conventions.md` + `code-review.md` + `git-commit.md` as scored baseline | Universal to every project. Domain files are project-specific. |
| All four agents are equal and independent | No dependency between setup guides. Each creates cold path if needed. |

## What M2.6 Does NOT Build

- `applyTo` frontmatter parsing or validation
- Content quality scoring (line count, duplication, specificity)
- Auto-generation of domain files from codebase analysis
- Per-directory `*/CLAUDE.md` or `*/GEMINI.md` scoring
- Copilot Chat integration testing (human gate covers this)
- Automatic migration of existing `.github/instructions/` files (migration guide is manual)
